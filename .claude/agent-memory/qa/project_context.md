**Architecture:** MVVM + Hilt + Room + Jetpack Compose + Navigation Component.

**Review chain:** dev implements → qa reviews → pm creates issues → Ruhan approves.

**Key conventions to verify in reviews:**
- ViewModels must be @HiltViewModel with @Inject constructor and SavedStateHandle for screens that receive nav args
- StateFlow collected in Composables should use collectAsStateWithLifecycle (lifecycle-runtime-ktx 2.8.7 provides this via lifecycle-runtime-compose artifact — NOTE: this artifact is NOT in libs.versions.toml, only lifecycle-viewmodel-compose is; project currently uses plain collectAsState())
- WebViews must be destroyed in DisposableEffect to prevent memory leaks
- Chip filters in Library use multi-select but DAO search() only takes a single muscleFilter string — multi-select beyond 1 is client-side only

**Phase 2 bugs found (2026-03-31):**
- CRITICAL: EditExercise route in NavHost does not pass exerciseId to AddExerciseViewModel via SavedStateHandle — backStackEntry not forwarded
- HIGH: Multi-chip filter broken — DAO search() takes single string; LibraryViewModel only passes muscles.first() for 2+ selections, then does in-memory filter; but the in-memory filter only runs when muscles.size > 1, meaning single-chip selection hits DB correctly but the in-memory pass is skipped — consistent behavior but the DB query itself uses substring LIKE which can cause false positives (e.g. "Back" matches "Full Body Back")
- HIGH: Instructions field is editable and "Save instructions" button is visible for seeded (isCustom=false) exercises — spec says core fields should be read-only
- HIGH: saveInstructions() in ExerciseDetailViewModel calls repository.update() which calls @Update DAO — this will silently succeed for seeded exercises, mutating read-only data
- MEDIUM: Converters.kt uses comma as delimiter for muscle group lists with no escaping — a value containing a comma (e.g. custom equipment "Barbell, Dumbbell") would corrupt the stored list on round-trip
- MEDIUM: collectAsState() used instead of collectAsStateWithLifecycle in LibraryScreen, ExerciseDetailScreen, AddExerciseScreen — lifecycle-runtime-compose not in dependencies
- MEDIUM: YouTubeWebView AndroidView has no DisposableEffect to call webView.destroy() — memory leak on back navigation
- MEDIUM: DifficultyBadge color matching uses exercise.difficulty.name but the when() cases are hardcoded strings ("Beginner" etc) — fragile but currently correct
- LOW: JSON import — when 0 valid entries parsed, summary dialog still shows (correct), but importedCount and skippedCount reset correctly; no bug here
- LOW: extractYouTubeId fallback calls input.takeLast(11) on unrecognized URLs — silently stores garbage as a video ID instead of returning null
- LOW: LibraryScreen uses Icons.Default.Upload which maps to a generic upload icon, not a JSON-specific indicator — UX only, not a bug

**Phase 4 bugs found (2026-03-31):**
- HIGH: collectAsStateWithLifecycle used in ALL Phase 4 screens (SearchYouTubeScreen, SessionScreen, SessionDetailScreen, HistoryScreen, PlanDetailScreen, WorkoutGeneratorNavHost) but lifecycle-runtime-compose artifact is still not in libs.versions.toml or build.gradle.kts — compile failure on those imports
- HIGH: ExerciseDetailScreen still uses plain collectAsState() (not fixed in Phase 4) — pre-existing but now more notable as new screens use withLifecycle
- HIGH: SessionViewModel.finishWorkout() calls workoutSessionDao.getById(state.sessionId).firstOrNull() — if the user taps "Finish Workout Early" before the init coroutine completes (sessionId is still ""), getById("") returns null and the ?: return@launch guard silently does nothing; session is never marked complete and finishCompleted is never set to true, so the user is stuck on the screen
- HIGH: HistoryViewModel uses flatMapLatest + inner flow{} for sessionSetDao.countBySessionId (suspend call inside flow builder) — this is fine at runtime but creates a new flow emission for EVERY upstream combined emission (every DB change to any plan or session triggers a full sequential count re-query for all sessions); no crash but causes quadratic DB load under many sessions
- MEDIUM: NetworkModule base URL is "https://www.googleapis.com/youtube/v3/" but YouTubeApiService.search() uses @GET("search") — Retrofit appends relative paths to the base URL, giving the correct final URL "https://www.googleapis.com/youtube/v3/search". This is CORRECT. (Not a bug — confirmed correct.)
- MEDIUM: SearchYouTubeViewModel.confirmVideoId() calls exerciseRepository.update(exercise.copy(youtubeVideoId = videoId)) — ExerciseRepository.update() maps Exercise domain → ExerciseEntity and calls @Update DAO. For seeded (isCustom=false) exercises this silently mutates read-only data — same pre-existing issue from Phase 2 (saveInstructions) but now also reachable via the YouTube link flow for ALL exercises.
- MEDIUM: SessionViewModel init creates a WorkoutSession row in the DB before the user has done anything. If the user presses Back before logging a single set, a ghost session row with completedAt=null is left in the DB. HistoryViewModel filters to completedAt != null so it won't appear in Sessions tab, but it's orphaned data. No cascade delete is triggered.
- MEDIUM: WorkoutGeneratorNavHost BadgedBox badge count uses historyUiState.plans.size — this counts ALL plans, not just new/unread ones. The badge never clears, making it permanently non-zero once any plan exists. This is a misleading UX signal.
- LOW: SessionScreen progress text shows "0/N exercises" on first set (currentExerciseIndex starts at 0); after completing all sets for an exercise, index increments to 1 and shows "1/N". The label is "exercises done" effectively, which is correct, but the progress bar reaches 100% only after the last exercise is indexed past the end (exercisesDone=true path), at which point the bar snaps to coerceIn(1f,1f) = 1f. No crash, but progress bar is misleading on the final exercise.
- LOW: formatDuration in both HistoryScreen and SessionDetailScreen has duplicate code — not a bug, just duplication.

**Phase 3 bugs found (2026-03-31):**
- CRITICAL: PPL type mismatch — pushMuscles/pullMuscles/legMuscles are Set<String> (MuscleGroup constants), but selectedMuscles is List<String>; `selected.contains(MuscleGroup.CORE)` works, but `pushMuscles.filter { it in selected }` compares String to String — actually fine at runtime BUT buildPplSessions receives List<String> from input.muscleGroups which is built from selectedMuscleGroups (Set<String> of MuscleGroup constant values). Actually compiles; low-level issue is that Full Body muscle is in MuscleGroup.ALL but NOT in any PPL bucket — selecting "Full Body" silently produces no exercises (falls through all three buckets, no Core either, returns empty list)
- CRITICAL: GenerateScreen.kt is still the placeholder stub ("Generate — coming in Phase 3") with no button or navigation trigger — the onStartGenerate callback is never called; users cannot reach GenerateFilterScreen from the bottom nav Generate tab
- HIGH: savePlan() never transitions _uiState to Saved — after insert completes it calls onSaved(planId) but leaves the state as Success; the Saved branch in the UI shows a spinner with no escape path. Since navigation is handled via the callback this is recoverable at runtime, but the Saved state is dead code and the Success state stays rendered behind the navigation transition
- HIGH: PlanDetailViewModel fetches exercise details by calling exerciseDao.getById(ex.exerciseId).first() inside a flow {} builder — this collects a Flow inside a flow builder using .first(), which is a suspend call inside a non-suspend lambda. This compiles (flow builder is a suspend context) but the inner flow is collected once per emission of the outer flatMapLatest; if any exercise is deleted between plan load and detail fetch, the entire flow crashes with NoSuchElementException from .first() on an empty flow (getById returns Flow<ExerciseEntity?> which can complete empty)
- HIGH: GenerateFilterScreen preferences pre-population is incomplete — GenerateFilterViewModel.init only restores exercisesPerMuscle from Room; daysPerWeek, selectedMuscleGroups, and splitType are not restored from any persisted state, so those fields always start at defaults (3 days, no muscles, SplitType.A) on every screen entry
- MEDIUM: GeneratedPlanScreen LaunchedEffect(input) — key is the GenerationInput? from sharedViewModel. On first composition input is null (sharedViewModel starts with null), so generate() is not called. Then input emits the real value and generate() fires. If the user presses back and returns to GeneratedPlan, LaunchedEffect will NOT re-fire (same input reference in the shared VM), so the plan stays visible — this is actually correct behavior. However if the user regenerates on GeneratedPlanScreen, then navigates back to GenerateFilter and submits the same input object again (same data class equality), LaunchedEffect will NOT re-fire and no new generation occurs. The plan silently shows the previous result.
- MEDIUM: Full Body muscle group in MuscleGroup.ALL is not mapped in any PPL bucket (push/pull/legs/core). Selecting only "Full Body" with PPL split returns an empty plan with no user-facing error — the engine returns GeneratedPlan with days=emptyList() and the UI shows "No exercises found" with no explanation that Full Body is incompatible with PPL.
- MEDIUM: WorkoutPlanExerciseEntity has ForeignKey RESTRICT on exercise_id — if a seeded exercise is soft-deleted (is_deleted=1) after being saved into a plan, the plan record is preserved (soft delete doesn't remove the row), so display falls back to ex.exerciseId as the name in PlanDetailScreen. This is a UX degradation but not a crash.
- MEDIUM: HistoryViewModel calls workoutPlanDao.getAll() which orders by created_at DESC at the DB level — sorting is correct. However plans list is exposed as StateFlow<List<WorkoutPlanEntity>> and the screen renders it directly; there is no empty-state guard in the StateFlow initial value path before the first DB emission (initialValue = emptyList()), so on first load a flash of "No saved plans yet" appears before real data arrives — minor visual glitch.
- LOW: GenerateFilterScreen shows 7 FilterChips for days-per-week in a single Row with no wrapping — on narrow screens (< ~360dp) chips will overflow horizontally and be clipped.
- LOW: GeneratedPlanScreen ExerciseRow onVideoClick lambda: `{ exercise.youtubeVideoId?.let { onExerciseDetail(exercise.exerciseId) } }` — the null check on youtubeVideoId is redundant since the IconButton for video is only shown when youtubeVideoId != null, but the outer lambda still captures exercise by closure correctly; no bug, just dead null check.
- LOW: PlanDetailScreen passes planId as a Composable parameter but PlanDetailViewModel reads it independently from SavedStateHandle — the planId parameter on the Composable is unused (only used for onStartWorkout). No functional bug since both read from the same nav argument, but the parameter is misleading.
